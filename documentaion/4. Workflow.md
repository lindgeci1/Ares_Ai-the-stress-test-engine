# ARES AI — User & System Workflow

## Overview

ARES AI is a document resilience testing platform that allows users to upload documents, perform multi-round audits using AI analysis, generate audio debates, and iteratively improve document quality through resilience scoring and AI-driven vulnerability detection.

---

## 1. Authentication Workflow

### 1.1 User Registration
```
User visits landing page
        ↓
"Sign Up" → Opens AuthModal with email + password form
        ↓
POST /auth/register (email, password, operator_name optional)
        ↓
Backend (user_handler.Register):
  - Validates email format & uniqueness
  - Hashes password with bcrypt
  - Creates user record in `users` table (status='active', subscription_tier='Free')
  - Auto-creates `user_usages` record (audit_limit=10, audits_performed=0)
  - Auto-assigns "User" role via user_roles junction
        ↓
Returns: {userId, email, subscription_tier, accessToken, refreshToken}
        ↓
Frontend stores tokens in sessionStorage, redirects to Dashboard
```

### 1.2 User Login
```
User visits Auth page
        ↓
"Log In" → Opens AuthModal with email + password form
        ↓
POST /auth/login (email, password)
        ↓
Backend (user_handler.Login):
  - Finds user by email
  - Compares hashed password with bcrypt
  - Generates JWT accessToken (15min) + refreshToken (7d)
  - Updates last login activity
        ↓
Returns: {userId, email, subscription_tier, accessToken, refreshToken}
        ↓
Frontend stores tokens, redirects to Dashboard
```

### 1.3 Token Refresh
```
AccessToken expires (15min expiry)
        ↓
Axios intercepts 401 response
        ↓
POST /auth/refresh (refreshToken from sessionStorage)
        ↓
Backend (user_handler.Refresh):
  - Validates refreshToken (must exist in refresh_tokens table, not expired)
  - Issues new accessToken + refreshToken pair
  - Invalidates old refreshToken
        ↓
Frontend retries original request with new accessToken
```

### 1.4 Logout
```
User clicks "Sign Out"
        ↓
Frontend clears tokens from sessionStorage
        ↓
Redirects to Landing page
        ↓
Optional: POST /auth/logout (backend invalidates refresh_tokens record)
```

## 2. Subscription & Payment Workflow

### 2.1 Subscription Tiers
```
Free Tier (default):
  - 10 audits per month
  - Basic features
  - Single audio debate per document

Pro Tier:
  - Unlimited audits
  - Advanced resilience metrics
  - Up to 5 rounds per document
  - Priority support

Enterprise Tier:
  - Custom audit limits
  - White-label options
  - Dedicated account manager
```

### 2.2 Payment Flow
```
User navigates to Billing page
        ↓
GET /admin/offers (frontend retrieves subscription packages)
        ↓
User selects offer → redirects to Checkout page
        ↓
Checkout page displays:
  - Selected offer details (price, features, tier_label)
  - User email pre-filled
  - Stripe payment form
        ↓
User clicks "Pay" / card details entered
        ↓
Frontend calls Stripe API (client-side)
        ↓
Stripe returns session_id or token
        ↓
POST /payments/create-payment (session_id, offer_id, email)
        ↓
Backend (payment_handler.CreatePayment):
  - Creates payment record in `payments` table (status='pending')
  - Links to offer_id, updates user.stripe_customer_id
  - Calls Stripe backend webhook handler
        ↓
Stripe webhook (payment_handler.HandleStripeWebhook):
  - Listens for checkout.session.completed event
  - Updates payment status='succeeded'
  - Updates user subscription_tier based on offer tier
  - Resets user_usages.audits_performed = 0 for Pro/Enterprise
        ↓
Frontend navigates to success page
        ↓
User subscription immediately upgraded
```

## 3. Document Upload & Processing Workflow

### 3.1 Document Upload
```
User on Dashboard clicks "Upload Document"
        ↓
NewAudit page → Document upload form (file picker)
        ↓
User selects file (PDF/TXT/DOCX) + optional metadata
        ↓
Frontend validates:
  - File size < 50MB
  - File type supported
  - User not exceeding audit_limit (user_usages.audits_performed)
        ↓
Frontend uploads to Cloudinary via Cloudinary SDK (client-side)
        ↓
Cloudinary returns cloudinary_url
        ↓
POST /documents/upload (file_name, cloudinary_url, raw_text optional)
        ↓
Backend (document_handler.UploadDocument):
  - Extracts text from cloudinary_url or raw_text param
  - Creates document record in `documents` table:
    * user_id = authenticated user ID
    * file_name = uploaded name
    * cloudinary_url = returned URL
    * raw_text = extracted text content
    * status = 'pending'
    * rounds_used = 1
  - Increments user_usages.audits_performed by 1
        ↓
Returns: {documentId, fileName, status, createdAt, roundsUsed}
        ↓
Frontend navigates to AuditLab page with documentId
```

### 3.2 Document Status States
```
pending    → Document uploaded, waiting for first audit analysis
processing → Audit analysis running (AI inference)
completed  → Audit report generated successfully
failed     → Audit or audio generation failed (retry available)
```

## 4. Audit & Resilience Analysis Workflow (Multi-Round)

### 4.1 First Audit (Round 1)

```
User on AuditLab page, document uploaded
        ↓
User clicks "Start Audit" or "Analyze Resilience"
        ↓
Frontend updates document status = 'processing'
        ↓
POST /documents/{id}/audit
        ↓
Backend (document_handler.AuditDocument):
  - Fetches document raw_text
  - Calls Ollama API (HTTP POST to Ollama_HOST/api/generate)
    * Prompt: "Analyze this document for logical fallacies, vulnerabilities, and argument resilience."
    * Model: mistral (or configured model)
    * Returns: AI-generated analysis JSON
  
  - Parses Ollama response into structured data:
    * vulnerabilities = extracted threat list (array of objects)
    * logical_fallacies = extracted fallacy list
    * fortification_plan = AI-suggested improvements
    * resilience_score = calculated 0-100 score
    * heatmap_data = threat distribution map (if applicable)
  
  - Creates AuditReport record in `audit_reports` table:
    * document_id = documents.id
    * round_number = 1
    * resilience_score = calculated score
    * all JSON fields populated
    * created_at = now
  
  - Updates documents.status = 'completed'
        ↓
Returns: {auditReport with round_number, resilience_score, vulnerabilities, etc.}
        ↓
Frontend (AuditLab page) renders:
  - Resilience Score card (0-100, color-coded)
  - Vulnerabilities list
  - Logical Fallacies list
  - Fortification Plan recommendations
  - Action buttons: "Generate Audio Debate", "Refine Document", "View Round History"
```

### 4.2 Subsequent Audits (Round 2+)

```
User improves document based on Round 1 feedback
        ↓
User reuploads/modifies document in same DocumentID (PUT /documents/{id})
        ↓
User clicks "Audit Again" (Round 2)
        ↓
POST /documents/{id}/audit?round=2
        ↓
Backend (document_handler.AuditDocument):
  - Fetches latest document raw_text
  - Increments documents.rounds_used = old_value + 1
  - Calls Ollama API with updated text
  - Creates NEW AuditReport record with round_number = 2
  - Same JSON field population as Round 1
        ↓
Frontend shows Round 2 results
        ↓
May display:
  - Comparison to Round 1 (score delta, new vs. fixed vulnerabilities)
  - Audit History modal showing all rounds with scores, trends
```

### 4.3 Resilience Score Interpretation
```
Score Range    | Tier        | Interpretation
0-20           | Critical    | Document has severe vulnerabilities (red)
21-40          | Poor        | Low resilience, major issues (orange)
41-60          | Fair        | Moderate vulnerience, needs work (yellow)
61-80          | Good        | Strong resilience with minor gaps (light green)
81-100         | Excellent   | Highly resilient, well-argued (dark green)
```

## 5. Audio Debate Generation Workflow

### 5.1 Audio Debate Creation

```
User on AuditLab, views completed AuditReport
        ↓
User clicks "Generate Audio Debate" button
        ↓
Frontend calls POST /documents/{id}/generate-audio
        ↓
Backend (document_handler.GenerateAudio):
  - Fetches latest AuditReport for this document
  - Extracts round_number from latest report
  - Calls Ollama API:
    * Input: document text + latest audit vulnerabilities
    * Prompt: "Generate a debate script where Pro speaker defends the document, Con speaker challenges it."
    * Returns: debate_script (JSON with speaker turns, arguments, counter-arguments)
  
  - Calls Google Cloud Text-to-Speech API (HTTP):
    * Pro voice: generated audio file (speaker 1)
    * Con voice: generated audio file (speaker 2)
    * Combines: merges Pro + Con audio into single MP3
    * Returns: audio_url
  
  - Uploads combined audio to Cloudinary
  
  - Creates AudioDebate record in `audio_debates` table:
    * document_id = documents.id
    * round_number = extracted from latest AuditReport
    * cloudinary_audio_url = Cloudinary URL
    * transcript_json = debate_script (for reference)
    * created_at = now
        ↓
Returns: {audioDebate with id, roundNumber, cloudinaryAudioURL, transcript}
        ↓
Frontend (AuditLab page) renders:
  - Audio Player (HTML5 <audio> with controls)
  - Transcript viewer (toggleable)
  - Download button (via Cloudinary URL)
  - Action: "Generate Another Round", "Share Audio", "View All Debates"
```

### 5.2 Multi-Round Audio Debates

```
User completes Round 2 audit (improved resilience score)
        ↓
User clicks "Generate Audio Debate for Round 2"
        ↓
Backend creates NEW AudioDebate record:
  - document_id = same
  - round_number = 2 (linked to Round 2's AuditReport)
  - Different audio content (based on Round 2 improvements)
        ↓
Frontend displays:
  - Debate audio player for Round 2
  - Transcript for Round 2
  - Side-by-side comparison option (Round 1 vs. Round 2 debates)
```

## 6. Admin Dashboard & Monitoring Workflow

### 6.1 Admin Stats & Metrics

```
Admin user logs in
        ↓
Navigates to AdminDashboard
        ↓
GET /admin/stats
        ↓
Backend (admin_handler.GetAdminStats):
  - Counts total_documents (all documents in DB)
  - Counts active_users (users with status='active')
  - Counts total_users (all users)
  - Counts total_audit_reports (all audit_reports)
  - Counts total_audio_debates (all audio_debates)
  - Calculates avg_resilience_score (AVG of non-null resilience_scores)
  - Counts processed_docs (documents with status='completed')
  - Counts pending_docs (documents with status='pending')
  - Counts failed_docs (documents with status='failed')
  - Aggregates total_payments (COUNT of payments)
  - Aggregates total_revenue (SUM of amount_paid for succeeded payments)
  - Fetches recent_activity (last 10 documents with user email, status, filename, timestamp)
        ↓
Returns JSON: {
  total_documents,
  active_users,
  total_users,
  total_audit_reports,
  total_audio_debates,
  avg_resilience,
  processed_docs,
  pending_docs,
  failed_docs,
  total_payments,
  total_revenue,
  recent_activity: [{id, fileName, status, createdAt, userEmail}, ...]
}
        ↓
Frontend (AdminDashboard) renders:
  - 6 metric cards (metrics row 1, row 2)
  - Recent activity table (live feed of latest documents)
  - Platform breakdown chart (processed % / pending % / failed %)
  - Subscription packages section (offers toggle)
```

### 6.2 Admin Document Management

```
Admin clicks "Documents" page
        ↓
GET /admin/documents (retrieves all documents from DB, not just own)
        ↓
Frontend displays table:
  - Columns: DOCUMENT, OWNER, RESILIENCE, STATUS, UPLOADED, ROUNDS, ACTIONS
  - Shows resilience % from latest AuditReport per document
  - Shows rounds_used count
  - Sortable by name, owner, resilience, date, rounds
        ↓
IF document has multiple audit reports (rounds_used > 1):
  - Clicking resilience % opens "Audit History" modal
  - Modal displays all rounds in DESC order (newest first)
  - Each round shows: Round #, Resilience Score %, Status, Threat Count, Color Bar
        ↓
Admin can delete/export documents via Actions menu
```

### 6.3 Admin Audio Debates Management

```
Admin clicks "Audio Debates" page
        ↓
GET /admin/audio-debates
        ↓
Backend (document_handler.GetAllAudioDebates):
  - Queries all AudioDebate records
  - Ordered by created_at DESC
  - Returns full records (JOINs with Document & User if needed)
        ↓
Frontend (AdminAudioDebates) enriches data:
  - Fetches all documents and users
  - Maps AudioDebate → AudioDebateRow:
    * id, documentId, roundNumber
    * documentName, ownerName, ownerEmail
    * cloudinaryAudioURL, createdAt
        ↓
Renders table:
  - Columns: ID, DOCUMENT (name + ID), ROUND, OWNER, CREATED, AUDIO URL
  - Searchable by: ID, document ID, round, document name, owner, email, URL
  - Each audio URL is clickable (opens player or download link)
```

## 7. Reset Password Workflow

### 7.1 Forgot Password Email

```
User on Auth page clicks "Forgot Password?"
        ↓
Redirects to ForgotPassword page with email input
        ↓
User enters email
        ↓
POST /auth/forgot-password (email)
        ↓
Backend (reset_password_handler.ForgotPassword):
  - Finds user by email (if not found, still returns success for privacy)
  - Generates reset_token (random, 32 chars)
  - Creates ResetPasswordRequest record in `reset_password_requests` table:
    * user_id = found user ID
    * token = hashed token
    * expires_at = now + 24 hours
  - Calls email_service.SendPasswordResetEmail(user.email, token)
  - Email service (SMTP or SendGrid):
    * Generates reset link: https://frontend.url/reset-password?token=TOKEN
    * Sends email with link
        ↓
Returns: {message: "Check your email for reset instructions"}
        ↓
User checks email, clicks reset link
```

### 7.2 Reset Password

```
User clicks reset link in email
        ↓
Browser opens /reset-password?token=TOKEN
        ↓
Frontend (ResetPassword page):
  - Extracts token from URL param
  - Displays password input form
        ↓
User enters new password
        ↓
POST /auth/reset-password (token, new_password)
        ↓
Backend (reset_password_handler.ResetPassword):
  - Finds ResetPasswordRequest by token (compare hashed token)
  - Validates expires_at > now (not expired)
  - Finds associated user
  - Hashes new_password with bcrypt
  - Updates user.password_hash
  - Deletes ResetPasswordRequest record
        ↓
Returns: {message: "Password reset successful"}
        ↓
Frontend redirects to Login page
        ↓
User logs in with new password
```

## 9. Error Handling & Recovery

### 9.1 Common Error Scenarios

```
Scenario 1: Audit Limit Exceeded
  → User on Free tier, performed 10 audits
  → Tries to audit 11th document
  → Backend returns: {error: "Audit limit reached, upgrade to Pro"}
  → Frontend shows upgrade prompt
  → User navigates to Billing page

Scenario 2: Token Expired During Request
  → User idle for 15+ minutes (accessToken expires)
  → Makes API request
  → Backend returns 401 Unauthorized
  → Axios interceptor catches 401
  → Calls POST /auth/refresh automatically
  → If successful: retries original request
  → If refresh fails: redirects to Login page

Scenario 3: Document Upload Fails
  → Cloudinary upload error
  → Frontend catches error, displays: "Upload failed, please try again"
  → User retries upload

Scenario 4: Audit Generation Fails
  → Ollama API unavailable or timeout
  → Backend returns {error: "Analysis failed"}
  → Frontend displays error toast
  → Document status remains 'pending' (not 'completed')
  → User can retry audit
```

## 10. Data Integrity & Cascades

### 10.1 Delete Operations

```
User deletes their account
        ↓
DELETE /users/{id}
        ↓
Backend:
  - Deletes user record (CASCADE deletes):
    * refresh_tokens (user_id FK)
    * user_usages (user_id FK)
    * payments (user_id FK)
    * documents (user_id FK, which CASCADE deletes):
      - audit_reports (document_id FK)
      - audio_debates (document_id FK)
        ↓
All user data and related records removed from DB
```

### 10.2 Document Deletion

```
User/Admin clicks delete on document in AdminDocuments
        ↓
DELETE /documents/{id}
        ↓
Backend:
  - Deletes document record (CASCADE deletes):
    * audit_reports (document_id FK)
    * audio_debates (document_id FK)
  - Optionally: calls Cloudinary API to delete document + audio files
        ↓
Document and all related audits/debates removed
```

## 11. System Performance & Caching

### 11.1 Frontend Caching (dataCache.ts)

```
AdminDashboard on mount:
  → Calls getAdminStats() + getAllOffers()
  → Stores results in dataCache (sessionStorage-backed)
  → Renders UI
  → On user action: refresh available via "Reload Stats" button
  → Manual refresh: clears cache + refetches

AdminDocuments on mount:
  → Calls getAllDocuments() + getUsers()
  → Caches all docs in dataCache
  → On sort/filter: uses cached data (front-end only)
  → Manual refresh: clears cache + refetches
```

### 11.2 Database Indexing

```
Indexed columns for query performance:
  - documents.user_id (FK lookup, fast retrieval per user)
  - audit_reports.document_id (fetch audits per document)
  - audio_debates.document_id (fetch debates per document)
  - users.email (login lookup)
  - refresh_tokens.user_id (token validation)
  - payments.user_id (payment history per user)
```

## 12. Summary: End-to-End User Journey

```
New User
  ├─ Lands on Landing page
  ├─ Signs Up (Register flow) → receives Free tier
  ├─ Logs In → Dashboard
  └─ New Audit page

Document Audit Lifecycle
  ├─ Upload document → creates Document (status='pending')
  ├─ Start audit (Round 1) → calls Ollama → creates AuditReport (round=1)
  ├─ View results → resilience score, vulnerabilities, recommendations

  │  ├─ Refine document → reupload
  │  ├─ Audit Again (Round 2) → new AuditReport (round=2)

  ├─ User on Billing page
  ├─ Selects Pro/Enterprise tier
  ├─ Redirects to Checkout

  ├─ Webhook confirms → user.tier updated

Admin Monitoring
  ├─ Admin logs in (auto-redirect to AdminDashboard)
  ├─ Views live metrics (/admin/stats)
  ├─ Manages documents (/admin/documents)
  ├─ Reviews audio debates (/admin/audio-debates)
  └─ Can delete/export any record (admin-only)
```

## 13. Technical Stack in Workflow Context

**Frontend Layer:**
- React components orchestrate user interactions → API calls
- Axios + JWT interceptors handle auth token refresh automatically
- React Router enforces ProtectedRoute (require login) vs. PublicRoute
- Tailwind CSS renders responsive UI across components

**Backend Layer:**
- Fiber HTTP handlers receive requests
- Middleware (RequireRole, auth) validates permissions
- Services contain business logic (audit generation, email sending)
- Repository layer queries GORM ORM
- GORM manages DB transactions and CASCADE deletes

**External Services:**
- Ollama API (HTTP) → AI analysis + debate generation
- Google Cloud TTS (HTTP) → dual-voice audio synthesis
- Cloudinary SDK → document + audio file storage
- Stripe SDK → payment processing
- SMTP/SendGrid → email delivery (password resets, notifications)

**Database (PostgreSQL on Supabase):**
- 10 main tables (users, roles, user_roles, documents, audit_reports, audio_debates, offers, payments, etc.)
- Foreign key constraints with CASCADE delete maintain data integrity
- Indexes on frequently queried columns (user_id, document_id) improve query performance
- JSONB columns store complex data (vulnerabilities, heatmap, transcripts)
